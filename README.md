# Kubeharness

A deterministic CLI plus agent-CLI wiring templates for Helm/Docker
deployments on Kubernetes. The Python package owns reproducible side
effects (lint, build, push, upgrade, wait, smoke); your agent CLI
(Claude Code, Codex CLI, ‚Ä¶) owns the reasoning and the loop.

See `docs/design.md` for the architecture rationale.

## Install

```bash
pip install kubeharness          # or: pip install -e . from a clone
```

### Prerequisites

The CLI shells out to the following tools. If one is missing the
check that uses it is marked `skip`, not `fail` ‚Äî so install only
what you actually need.

| Tool | Used for |
|---|---|
| `helm` (v3) | chart lint, template, upgrade, uninstall |
| `kubectl` | post-deploy wait, pod inspection |
| `docker` (+ buildx) | build + push images |
| `yamllint` | chart YAML hygiene (shipped via pip dep) |
| `kubeconform` | k8s schema validation (`helm template | kubeconform`) |
| `hadolint` | Dockerfile lint |
| `trivy` | chart config scan (optional) |
| `gitleaks` | secret scan (optional) |

Install guidance lives with each upstream project; Kubeharness does
not ship an installer script.

## Quickstart

```bash
# 1. Scaffold a new consumer project (or seed an existing one).
mkdir my-infra && cd my-infra
python -m harness init --name my-infra --workspace workspace

# 2. Fill out config/harness.yaml ‚Äî every TODO(init) is a value you
#    must change before first deploy (cluster.namespace, registry,
#    environments.*).
$EDITOR config/harness.yaml

# 3. Put service artifacts under the workspace dir.
mkdir -p workspace/helm/prometheus workspace/docker/prometheus
# ... author Chart.yaml, values.yaml, Dockerfile, ...

# 4. Run checks directly, or via your agent CLI.
python -m harness verify-static  --service prometheus
python -m harness apply          --service prometheus
python -m harness verify-runtime --service prometheus \
    --phase observability --sub-goal prometheus
```

Each command prints a single JSON envelope and exits:

- `0` everything passed
- `1` at least one check failed
- `2` config error (missing YAML, bad schema)

The full subprocess transcript lands in
`logs/deploy/<ts>-<service>-<stage>-standalone.log` (or the file set
by `$HARNESS_SESSION_LOG`, if exported before invocation).

## Agent CLI workflow

`init` also scaffolds a `.claude/` tree with one slash command
(`/deploy`), two subagents (`deploy-orchestrator`, `runtime-diagnoser`),
five skills, and permission/hook wiring. In Claude Code:

```
/deploy prometheus
```

runs verify-static ‚Üí apply ‚Üí verify-runtime with a retry budget;
runtime failures are diagnosed by the read-only `runtime-diagnoser`
subagent and proposed fixes are applied through the `ask` permission
(every write requires human approval). See `AGENTS.md` (generated by
`init`) for the full contract.

Codex CLI and other agent runners read the same `AGENTS.md`. The
`.claude/` tree is Claude-Code-specific wiring ‚Äî ignore it if you're
using a different CLI.

## Configuration

`config/harness.yaml` is the single source of truth. Key sections:

- `cluster` ‚Äî namespace, optional kubeconfig
- `conventions` ‚Äî workspace dir, chart/docker/smoke path patterns,
  release naming, image tag, registry, write allow/deny globs
- `environments.<env>` ‚Äî domain suffix, arch (for docker
  `--platform`), node selectors. `active` picks the env used by
  `apply` / `verify-runtime`
- `checks.static.*` / `checks.runtime.*` ‚Äî enable/disable flags and
  `kubectl_wait` timings
- `logging` ‚Äî session log dir, JSON `log_tail` size, retention
- `orchestration` ‚Äî `max_runtime_retries` budget for the orchestrator
  subagent

See `templates/config/harness.yaml.example.tmpl` (or the scaffolded
copy in your project) for the complete schema with inline comments.

## Development

```bash
pip install -e .[dev]
.venv/bin/pytest tests/ -v
```

Tests are hermetic ‚Äî every subprocess is stubbed by label. A full
run takes under two seconds and requires no cluster, no docker, and
no external CLIs.
